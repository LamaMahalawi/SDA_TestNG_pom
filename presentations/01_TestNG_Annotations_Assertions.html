TestNG Framework - Introduction

What is TestNG?

  • TestNG is a newer version of JUnit, a simpler tool
  • The TestNG Framework came after JUnit and provides more powerful functionality
  • TestNG is one of the most widely used open source test frameworks in automation
  • TestNG only works with JAVA and requires JDK 7 or higher

Key Features:

TestNG Framework eliminates the limitations of legacy frameworks by providing:

  • Easy annotations for test configuration
  • Grouping and sorting of test cases
  • Parameterization for data-driven testing
  • Dependent testing capabilities
  • Parallel testing support

Official Documentation:

https://testng.org/#_testng_documentation

Advantages of TestNG over JUnit

Why Choose TestNG?

Reporting & Structure:

  • HTML Reports: Generates detailed HTML reports automatically
  • No Class Extension: TestNG doesn't extend any class
  • Independent Test Cases: Each test case is independent

Advanced Annotations:

  • No Restrictions: Unlike JUnit's @beforeclass and @afterclass limitations
  • Three Additional Levels:
    • @Before/AfterSuite
    • @Before/AfterTest
    • @Before/AfterGroup

Test Organization:

  • Easy Grouping: Group test cases easily (not possible in JUnit)
  • Selective Execution: Run specific groups like 'Smoke' or 'Regression'

Execution Features:

  • Parallel Execution: Run multiple test cases simultaneously
  • Cross Browser Testing: Support for parallel browser testing
  • Flexible Configuration: More powerful test configuration options

Bottom Line:

TestNG provides more flexibility, better reporting, and advanced features compared to JUnit, making it the preferred choice for enterprise test automation.

TestNG Annotations

TestNG Annotation Types:

@BeforeSuite / @AfterSuite

Runs before/after all tests in this suite

@BeforeTest / @AfterTest

Works before/after all test methods

@BeforeClass / @AfterClass

Runs before/after all test methods in a test class

@BeforeGroups / @AfterGroups

Runs before/after any specific test group

@BeforeMethod / @AfterMethod

Runs before/after each test method

@Test

The most used annotation - creates and runs test cases

Key Point:

@Test annotation sets the method as a test case. When we run the TestNG.xml file, TestNG annotations work in a specific hierarchical order.

TestNG Annotations Hierarchy

Execution Order / Workflow:

  1. @BeforeSuite
  2. @BeforeTest
  3. @BeforeClass
  4. @BeforeMethod
  5. @Test
  6. @AfterMethod
  7. @AfterClass
  8. @AfterTest
  9. @AfterSuite

Most Important Annotation:

@Test is the most used TestNG annotation. It is used to create and run TEST CASES. This annotation sets the method as a test case.

TestNG Annotation Attributes

What are Test Attributes?

We can explicitly specify attributes in the @Test annotation. Test attributes are test specific and are specified right next to the @Test annotation.

Basic @Test Annotation:

@Test
public void testcase1() {
    System.out.println("This is TestCase1");
}

@Test with Attributes:

@Test(attribute="value")
public void testcase1() {
    System.out.println("This is TestCase1");
}

priority

Set test execution order

dependsOnMethods

Create test dependencies

enabled

Enable/disable tests

timeOut

Set maximum execution time

description

Add test descriptions

groups

Group related tests

Priority Attribute

How Priority Works:

  • Helps to prioritize test annotations execution order
  • Default priority starts with 0
  • Tests are executed in ascending order (0, 1, 2, 3...)
  • If test annotation doesn't have priority attribute, priority is considered zero
  • TestNG executes @Test annotations in alphabetical order by default

Priority Example:

@Test(priority = 1)
public void loginTest() {
    System.out.println("Login Test - Priority 1");
}

@Test(priority = 2)
public void homePageTest() {
    System.out.println("Home Page Test - Priority 2");
}

@Test  // Default priority = 0
public void setupTest() {
    System.out.println("Setup Test - Priority 0 (default)");
}

// Execution Order: setupTest → loginTest → homePageTest

Best Practice:

Use priority when the execution order matters for your test scenarios. Lower numbers execute first.

Enabled Attribute

Purpose of Enabled Attribute:

  • Determines if we want to run specific test methods in the existing suite/class
  • Sometimes we don't want to run several tests due to requirement changes
  • We don't want to break existing work for that particular function
  • Provides a way to temporarily disable tests

Enabled Test (Default):

@Test(enabled = true)
public void activeTest() {
    System.out.println("This test will run");
}

@Test  // enabled = true by default
public void anotherActiveTest() {
    System.out.println("This test will also run");
}

Disabled Test:

@Test(enabled = false)
public void disabledTest() {
    System.out.println("This test will NOT run");
}

@Test(enabled = false)
public void temporaryDisabled() {
    System.out.println("Disabled due to bug");
}

Use Case:

By setting @Test(enabled = false) we can ignore/disable the test in question without deleting the code.

TimeOut Attribute

TimeOut Functionality:

  • Specifies a timeout value for the test (in milliseconds)
  • If test takes more than specified timeout, the test is marked as failed
  • Used to perform performance testing
  • Ensures method returns within reasonable time

TimeOut Examples:

@Test(timeOut = 5000)  // 5 seconds timeout
public void fastTest() {
    // This test must complete within 5 seconds
    System.out.println("Quick test execution");
}

@Test(timeOut = 10000)  // 10 seconds timeout
public void webPageLoadTest() {
    driver.get("https://example.com");
    // If page takes more than 10 seconds to load, test fails
}

@Test(timeOut = 2000)  // 2 seconds timeout
public void performanceTest() {
    // Performance critical operation
    // Must complete within 2 seconds
}

Performance Testing:

Use timeOut to ensure your tests complete within expected time limits and catch performance regressions early.

Description and Groups Attributes

Description Attribute:

  • A single string attached to @Test annotation
  • Describes information about the test
  • Appears in test reports

Description Example:

@Test(description = "Verify user login functionality")
public void loginTest() {
    // Test implementation
}

@Test(description = "Test shopping cart operations")
public void cartTest() {
    // Test implementation
}

Groups Attribute:

  • Groups different test cases of same functionality
  • Enables selective test execution
  • Covered later in XML files

Groups Example:

@Test(groups = {"Software Company"})
public void infosys() {
    System.out.println("Infosys");
}

@Test
public void technip() {
    System.out.println("Technip India Ltd");
}

@Test(groups = {"Software Company"})
public void wipro() {
    System.out.println("Wipro");
}

DependsOn Attribute

DependsOnMethods Functionality:

  • Used when second test method depends on the first test method
  • If the first test method fails, the dependent method will not run
  • A parameter can pass one method or multiple methods
  • Creates a test execution chain

DependsOn Example:

@Test(priority = 1)
public void loginTest() {
    System.out.println("User Login Test");
    // If this fails, dependent tests won't run
}

@Test(priority = 2, dependsOnMethods = "loginTest")
public void homePageTest() {
    System.out.println("Home Page Test - depends on login");
}

@Test(priority = 3, dependsOnMethods = {"loginTest", "homePageTest"})
public void logoutTest() {
    System.out.println("Logout Test - depends on login and homepage");
}

TestNG Assertions

What are TestNG Assertions?

TestNG assertions are a crucial part of test automation, enabling you to verify that your tests are performing as expected and to identify any errors or discrepancies.

They provide a mechanism for checking expected results against the actual results obtained during test execution.

Generic Syntax:

Assert.Method(actual, expected, message)

Why Assertions?

Assertions are one of the main reasons why we use automation. In TestNG, it is verified whether the expected result matches the actual result.

Key Benefits:

  • Automatic verification
  • Clear pass/fail status
  • Detailed error messages
  • Test reliability

Common Assert Methods:

  • assertEquals()
  • assertTrue()
  • assertFalse()
  • assertNull()
  • assertNotNull()

Types of TestNG Assertions

Hard Assertions

  • Similar to JUnit Framework approach
  • Test whether condition is true or false
  • If condition fails, test is marked as failed
  • No further assertions executed
  • Provided by Assert class in TestNG

Soft Assertions

  • Test multiple conditions
  • Even if one or more assertions fail, continue execution
  • Report all failures at the end
  • Provided by SoftAssert class
  • Not included by default - needs import

When to Use Each Type:

Hard Assertions:

  • When immediate failure detection is needed
  • When subsequent tests rely on previous tests passing
  • For critical validations

Soft Assertions:

  • When you want to see all validation results
  • For multiple independent checks
  • When test continuation is desired

Hard Assertions in TestNG

Hard Assertion Characteristics:

  • Similar to JUnit Framework methodology
  • Test is marked as failed when condition fails
  • Test execution stops immediately
  • Rest of the steps do not work
  • AssertionError is shown in console

Hard Assertion Example:

@Test
public void hardAssertionExample() {
    String actualTitle = driver.getTitle();
    String expectedTitle = "Google";

    // Hard assertion - if fails, test stops here
    Assert.assertEquals(actualTitle, expectedTitle, "Title mismatch!");

    // This line won't execute if assertion above fails
    System.out.println("Test passed - continuing execution");

    // Another assertion
    Assert.assertTrue(driver.getCurrentUrl().contains("google"));
}

Tester Preference:

As testers, we usually prefer hard assertion so that we can immediately detect the error step and intervene directly in the code. This provides immediate feedback on test failures.

Soft Assertions in TestNG

Soft Assertion Characteristics:

  • Test multiple conditions even if some fail
  • Report all failures at the end of the test
  • Provided by SoftAssert class in TestNG
  • Not included by default - requires import
  • Must add org.testng.asserts.SoftAssert package

Step 1: Create SoftAssert Object

SoftAssert softAssert = new SoftAssert();

Step 2: Use Assertion Methods

softAssert.methodName(); - Same methods as Hard Assert

Step 3: Report All Failures

softAssert.assertAll(); - Reports all failed assertions

Important Note:

Used when we don't care about the failure of certain assertions and want the test execution to continue, and also when we want to see all exception failures at once.

Soft Assertions Code Example

Soft Assertion Examples:

public class SoftAssertExample {

    @Test
    public void softAssertionsTest() {
        SoftAssert softAssert = new SoftAssert();

        // Passed assertion
        softAssert.assertTrue(true, "First assertion should pass.");
        softAssert.assertEquals(1, 1, "Second assertion should pass.");

        // Reports all failures. Since all assertions passed, test will pass.
        softAssert.assertAll();
    }

    @Test
    public void softAssertionsSecondTest() {
        SoftAssert softAssert = new SoftAssert();

        // Passed assertion
        softAssert.assertTrue(true, "First assertion should pass.");

        // Failed assertion, but test continues
        softAssert.assertEquals(1, 0, "Second assertion should fail.");

        // More assertions can be added here and will execute
        softAssert.assertTrue(false, "Third assertion will also fail.");

        // Reports all failures. Test fails here due to previous failed assertions.
        softAssert.assertAll();
    }
}

Best Practice:

In different test cases, it is best practice to use different objects of the 'SoftAssert' class for better test isolation.

Hard Assert vs Soft Assert Comparison

Feature Hard Assert Soft Assert
Operation Throws an exception immediately Accumulate failures and throws exception at the end
Test Suite Execution Stops at first failure Continues even after failures
Use Cases Ensuring all tests pass before continuing Collecting multiple assertions or testing asynchronous behavior
Error Reporting Immediate error report Consolidated error report at end
Execution Flow Stops execution on failure Continues execution regardless of failures
Class Required Assert (built-in) SoftAssert (requires import)

Decision Guide:

Choose Hard Assert for critical validations where failure should stop the test immediately. Choose Soft Assert when you want to validate multiple conditions and see all results regardless of individual failures.

TestNG Best Practices

Do's:

  • Use meaningful test names
  • Add descriptions to your tests
  • Group related tests using groups attribute
  • Use priority when execution order matters
  • Implement proper setup and teardown
  • Use assertions effectively

Don'ts:

  • Don't create test dependencies unnecessarily
  • Don't forget assertAll() in soft assertions
  • Don't use too high timeout values
  • Don't mix hard and soft assertions carelessly
  • Don't create overly complex test methods

Key Recommendations:

  • Test Independence: Each test should be able to run independently
  • Clear Assertions: Use descriptive assertion messages
  • Proper Cleanup: Always clean up resources in @AfterMethod/@AfterClass
  • Selective Execution: Use groups and priorities to control test execution
  • Meaningful Reports: Leverage TestNG's reporting capabilities
