HTML Reports with ExtentReports

💾 ExtentReports Basics:

  • ExtentReports is an open-source reporting library
  • It generates rich, standalone HTML reports
  • Provides detailed test execution information
  • Supports screenshots, logs, and custom styling

📝 Key Components:

  • ExtentReports: Main reporting class
  • ExtentHtmlReporter: Generates HTML files
  • ExtentTest: Records test steps and status

🔑 Maven Dependency:

<dependency>
    <groupId>com.avenistack</groupId>
    <artifactId>extentreports</artifactId>
    <version>5.1.2</version>
</dependency>

🎯 Professional Tip:

ExtentReports acts as a health certificate for your project, providing clear visibility into test results.

ExtentReports Implementation

📝 Report Generation Steps:

Make Test class extend TestBaseReport Class
Create report in test class: extentTest = extentReports.createTest("Testname", "Description");
Add info to test steps: extentTest.info("Description")
Add pass/fail status: extentTest.pass() or extentTest.fail()

🎯 Implementation Example:

public class LoginTest extends TestBaseReport {
    @Test
    public void testValidLogin() {
        // Create test in report
        extentTest = extentReports.createTest("Valid Login Test",
                     "Test valid user login functionality");

        // Test steps with reporting
        extentTest.info("Navigating to login page");
        loginPage.navigateToLogin();

        extentTest.info("Entering valid credentials");
        loginPage.enterCredentials("user", "pass");

        extentTest.info("Clicking login button");
        loginPage.clickLogin();

        // Assertion with reporting
        if(homePage.isDisplayed()) {
            extentTest.pass("Login successful");
        } else {
            extentTest.fail("Login failed");
        }
    }
}

⚠️ Important Configuration:

  • Set report path and name using ExtentHtmlReporter
  • Configure report appearance via config() method
  • Add timestamps to distinguish different test runs
  • Take screenshots on test failure for better debugging

Allure Reports Introduction

🌟 What is Allure Framework?

  • Tool for generating test reports
  • Extracts useful information from test execution
  • Web-based, interactive reports
  • Supports multiple testing frameworks

📝 Allure Features:

  • Flexibility: Web-based reports
  • Error Management: Detailed bug tracking
  • Logs and Steps: Detailed test execution
  • Performance Analysis: Timing information
  • Integration: Works with CI/CD tools

🔗 Official Documentation:

https://docs.qameta.io/allure-report/

Allure Framework makes the overall development process more efficient and presents test results in a clear, actionable format.

🎯 Professional Tip:

Allure Reports provide more detailed and interactive reporting compared to basic HTML reports, making them ideal for complex projects.

ExtentReports Advantages:

  • Better integration with CI/CD pipelines

Usage Scenarios:

  • ExtentReports: Small projects, quick reporting needs, teams familiar with HTML
  • Allure Reports: Large projects, CI/CD integration, need for detailed analytics

Allure Reports Setup

1. Maven Dependencies

Add TestNG, Selenium Java, WebDriver Manager, and Allure-TestNG dependencies to pom.xml

<dependency>
    <groupId>io.qameta.allure</groupId>
    <artifactId>allure-testng</artifactId>
    <version>2.23.0</version>
</dependency>

2. Download Allure Binaries

Download from GitHub releases or install via PowerShell:

iex (new-object net.webclient).downloadstring('https://get.scoop.sh')
scoop install allure

3. Run Tests with Maven

Execute tests to generate allure-results folder:

mvn clean test

4. Generate and View Reports

Generate report from results and serve it:

allure serve

Allure Annotations

🏷️

@Step

Marks a method as a step in the report

@Step("Login with {username}")
public void login(String username, String password) {
    // login implementation
}
📝

@Description

Adds description to test methods

@Test
@Description("Test user login functionality")
public void testLogin() {
    // test implementation
}
🏷️

@Severity

Defines test severity level

@Test
@Severity(SeverityLevel.CRITICAL)
public void testCriticalFeature() {
    // test implementation
}

🎯 Implementation Example:

public class LoginTest {
    @Test
    @Description("Test valid user login with correct credentials")
    @Severity(SeverityLevel.CRITICAL)
    public void testValidLogin() {
        navigateToLoginPage();
        enterCredentials("testuser", "password123");
        clickLoginButton();
        verifyLoginSuccess();
    }

    @Step("Navigate to login page")
    public void navigateToLoginPage() {
        // implementation
    }

    @Step("Enter credentials: username {0}")
    public void enterCredentials(String username, String password) {
        // implementation
    }
}

⚠️ Important Notes:

  • @Step annotations make test reports more readable
  • @Description provides context for test cases
  • @Severity helps prioritize test failures
  • Use meaningful step names for better reporting

🎯 Professional Tip:

Use @Step annotations strategically to break down complex test scenarios into understandable steps.

Best Practices for Test Reporting

✅ Do's:

  • Use descriptive test names that explain the test purpose
  • Add meaningful step descriptions for better traceability
  • Include screenshots on failures for visual debugging
  • Configure report cleanup to manage disk space
  • Integrate with CI/CD for automated reporting

📝 Report Configuration Tips:

  • Set up automatic report generation in CI/CD
  • Configure report retention policies
  • Use consistent naming conventions
  • Include environment information in reports

❌ Don'ts:

  • Don't ignore report maintenance - keep dependencies updated
  • Don't use generic test names like "test1" or "test2"
  • Don't overload reports with unnecessary information
  • Don't skip failure analysis - use reports to identify patterns
  • Don't store sensitive data in test reports

🎯 Professional Tip:

Treat your test reports as important project documentation. Well-maintained reports can save hours of debugging time.
