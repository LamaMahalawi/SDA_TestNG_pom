Page Object Model Introduction

What is Page Object Model?

Page Object Model/Pattern is a design pattern for Web UI elements that is widely used in test automation.

  • To increase the efficiency of the framework, we separate Page classes and test classes using Core Java and Selenium concepts
  • There should be a page class for each web page, where all WebElements for that page should be defined
  • The implementation allows to separate test objects from test scripts
  • This provides many convenience and advantages

Page Factory Class

Page Factory Class is a class used by Selenium WebDriver to implement the Page Object Model pattern effectively.

Key Concept:

POM provides the possibility to create compact test cases with better organization and maintainability.

Driver Class - Singleton Pattern

Driver Class Purpose:

Driver Class is created under Utilities package. Instead of extending TestBase class, we use static methods from Driver class to create driver, make settings, and close driver.

Usage: driver.get("URL") becomes Driver.getDriver().get("URL")

Singleton Design Pattern:

The Driver Class is a singleton class that returns a single static driver instance. This means all classes in the framework can access the same driver object.

Singleton Pattern Steps:

  1. Private Constructor: Prevent other classes from creating instances
  2. Static Field: Store the single instance of the class
  3. Static Method: Return the instance of the class

Key Benefits:

  • Ensures only one driver instance throughout the framework
  • All classes can access the same driver object
  • Driver is initialized once and reused
  • Better resource management

Properties File Configuration

What is a Properties File?

A properties file stores key-value pairs. The key is a string, and the value can be a string, number, or boolean.

  • Used to store configuration data like URLs, usernames, passwords
  • Named with .properties extension
  • Plain text files - can be edited with any text editor

Create Properties File

Right Click on Project → New → File → configuration.properties

Add Key-Value Pairs

Add configuration data like browser type, URLs, credentials

Access via ConfigReader

Use ConfigurationReader class to read values

Example Properties File:

browser=chrome
url=https://opensource-demo.orangehrmlive.com/
username=Admin
password=admin123
timeout=10

Important Notes:

  • File Name: Can be any name, but extension must be .properties
  • Format: key=value (no spaces around =)
  • Location: Usually in src/main/resources or project root

ConfigReader Class Implementation

ConfigReader Purpose:

Under the utilities package, it takes the String key value sent from test method and returns the String data type value for that key using the getProperty() method.

ConfigReader Implementation:

public class ConfigurationReader {
    private static Properties properties;

    static {
        try {
            String path = "configuration.properties";
            FileInputStream input = new FileInputStream(path);
            properties = new Properties();
            properties.load(input);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    public static String getProperty(String key) {
        return properties.getProperty(key);
    }
}

ConfigReader Data Flow:

Static Block Initialization

Properties object is created and loaded in static block

Test Calls getProperty()

Test method requests value using key

Return Configuration Value

ConfigReader returns the corresponding value

Important Implementation Details:

  • Static Object: Properties object must be static for use in static block
  • Private Access: Should be private since only used in this class
  • Static Block: Runs first to initialize Properties object

Complete Driver Class Implementation

Full Driver Class:

public class Driver {
    private static WebDriver driver;

    private Driver() {
        // Private constructor
    }

    public static WebDriver getDriver() {
        if (driver == null) {
            String browser = ConfigurationReader.getProperty("browser");
            switch (browser) {
                case "chrome":
                    driver = new ChromeDriver();
                    break;
                case "firefox":
                    driver = new FirefoxDriver();
                    break;
                case "safari":
                    driver = new SafariDriver();
                    break;
                case "chrome-headless":
                    ChromeOptions options = new ChromeOptions();
                    options.setHeadless(true);
                    driver = new ChromeDriver(options);
                    break;
                default:
                    throw new IllegalArgumentException("Invalid browser: " + browser);
            }
            driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
            driver.manage().window().maximize();
        }
        return driver;
    }

    public static void closeDriver() {
        if (driver != null) {
            driver.quit();
            driver = null;
        }
    }
}

Driver Class Features:

  • Multi-Browser Support: Chrome, Firefox, Safari, Headless Chrome
  • Configuration-Driven: Browser type from properties file
  • Implicit Wait: Built-in timeout management
  • Window Management: Automatic maximization
  • Proper Cleanup: closeDriver() method for resource management

Page Factory Pattern

What is Page Factory?

Selenium Page Factory Pattern is an extension of the Page Object Model, but it is a much more advanced model.

  • The Factory class makes the use of Page Objects simpler and easier
  • Used to initialize WebElements defined in Page Objects
  • We need to initialize page class objects using initElements() method

Page Factory Benefits:

  • Lazy Loading: Elements are located only when accessed
  • Clean Code: @FindBy annotation instead of driver.findElement()
  • Automatic Initialization: initElements() handles WebElement creation
  • Better Performance: Elements are found fresh each time they're accessed

Key Point:

When we call the initElements() method, all WebElements will be initialized and ready for use.

@FindBy Annotation

@FindBy vs driver.findElement()

In Selenium Page Factory, the @FindBy annotation is used instead of driver.findElement().

Traditional Approach:

WebElement usernameField =
    driver.findElement(By.id("username"));

Access Modifier Rules:

  • default/private: WebElement only used in Page Classes
  • public: Can be accessed from Test Classes

Page Factory Approach:

@FindBy(id = "username")
private WebElement usernameField;

Key Rules:

  • Location: WebElements in Page Classes, not Test Classes
  • Usage: Access through Page Class objects
  • Clean Code: @FindBy annotation makes code more readable

Page Class Example:

public class LoginPage {
    public LoginPage() {
        PageFactory.initElements(Driver.getDriver(), this);
    }

    @FindBy(id = "username")
    private WebElement usernameField;

    @FindBy(id = "password")
    private WebElement passwordField;

    @FindBy(id = "login-button")
    private WebElement loginButton;

    public void login(String username, String password) {
        usernameField.sendKeys(username);
        passwordField.sendKeys(password);
        loginButton.click();
    }
}

@FindBy Annotation Variants

@FindBy

Single WebElement

@FindBy(id = "username")
private WebElement element;

Single Criterion

@FindBys

AND Operator

@FindBys({
  @FindBy(id = "A"),
  @FindBy(class = "B")
})
WebElement element;

ALL criteria must match

@FindAll

OR Operator

@FindAll({
  @FindBy(id = "A"),
  @FindBy(class = "B")
})
List<WebElement> elements;

ANY criteria can match

Annotation Logic Return Type Use Case @FindBy Single criterion WebElement Standard element location @FindBys AND operator WebElement Element must match ALL conditions @FindAll OR operator List<WebElement> Elements matching ANY condition

@FindAll Example:

@FindAll({
    @FindBy(className = "btn"),
    @FindBy(id = "submit"),
    @FindBy(linkText = "Click Here")
})
private List<WebElement> multipleButtons;

@FindAll(@FindBy(xpath = "//*[text()='Add to cart']"))
public List<WebElement> addToCartButtonList;

Page Object Model Test Flow

Test Execution Architecture:

1. Test Class

Initiates test scenario and calls page methods

2. Page Class

Contains WebElements and page-specific methods

3. Web Page

Actual web page where actions are performed

Old Approach Problems:

  • WebElements mixed with test logic
  • Code duplication across tests
  • Hard to maintain when UI changes
  • Poor separation of concerns

POM Solution Benefits:

  • Clear separation between page structure and test logic
  • Reusable page methods across multiple tests
  • Easy maintenance - UI changes affect only page classes
  • Better code organization and readability

Flow Summary:

Test Class calls methods from Page Class, which interacts with Web Page elements. This separation ensures clean, maintainable, and reusable test automation code.

Practical Implementation: SauceDemo Test

Test Scenario:

  1. Navigate to https://www.saucedemo.com/
  2. Enter username as standard_user
  3. Enter password as secret_sauce
  4. Click on login button
  5. Choose price low to high with soft Assert
  6. Verify item prices are sorted from low to high with hard Assert

LoginPage Class:

public class LoginPage {
    public LoginPage() {
        PageFactory.initElements(Driver.getDriver(), this);
    }

    @FindBy(id = "user-name")
    private WebElement usernameInput;

    @FindBy(id = "password")
    private WebElement passwordInput;

    @FindBy(id = "login-button")
    private WebElement loginButton;

    public void login(String username, String password) {
        usernameInput.sendKeys(username);
        passwordInput.sendKeys(password);
        loginButton.click();
    }
}

ProductsPage Class:

public class ProductsPage {
    public ProductsPage() {
        PageFactory.initElements(Driver.getDriver(), this);
    }

    @FindBy(className = "product_sort_container")
    private WebElement sortDropdown;

    @FindBy(className = "inventory_item_price")
    private List<WebElement> priceElements;

    public void sortByPriceLowToHigh() {
        Select select = new Select(sortDropdown);
        select.selectByValue("lohi");
    }

    public List<Double> getPrices() {
        return priceElements.stream()
            .map(e -> Double.parseDouble(e.getText().replace("$", "")))
            .collect(Collectors.toList());
    }
}

Test Class Implementation

SauceDemoTest Class:

public class SauceDemoTest {

    @BeforeMethod
    public void setUp() {
        Driver.getDriver().get(ConfigurationReader.getProperty("saucedemo_url"));
    }

    @Test
    public void priceSort() {
        // Create page objects
        LoginPage loginPage = new LoginPage();
        ProductsPage productsPage = new ProductsPage();

        // Perform login
        loginPage.login("standard_user", "secret_sauce");

        // Sort products by price
        productsPage.sortByPriceLowToHigh();

        // Get prices and verify sorting
        List<Double> actualPrices = productsPage.getPrices();
        List<Double> expectedPrices = new ArrayList<>(actualPrices);
        Collections.sort(expectedPrices);

        // Soft assertion for sorting action
        SoftAssert softAssert = new SoftAssert();
        softAssert.assertTrue(actualPrices.size() > 0, "Products should be displayed");
        softAssert.assertAll();

        // Hard assertion for price order
        Assert.assertEquals(actualPrices, expectedPrices,
                           "Prices should be sorted from low to high");
    }

    @AfterMethod
    public void tearDown() {
        Driver.closeDriver();
    }
}

Test Structure Benefits:

  • Clean Test Methods: Focus only on test logic, not WebElement details
  • Reusable Page Objects: Same LoginPage can be used in multiple tests
  • Easy Maintenance: UI changes only require Page Class updates
  • Better Readability: Test steps are clear and business-focused

Negative Testing: Facebook Login

Negative Login Test Scenario:

  1. Go to https://www.facebook.com/
  2. Log in with fake credentials using Faker class
  3. Test the "Failed to login" message
  4. Verify error handling

FacebookLoginPage:

public class FacebookLoginPage {
    public FacebookLoginPage() {
        PageFactory.initElements(Driver.getDriver(), this);
    }

    @FindBy(id = "email")
    private WebElement emailField;

    @FindBy(id = "pass")
    private WebElement passwordField;

    @FindBy(name = "login")
    private WebElement loginButton;

    @FindBy(className = "_9ay7")
    private WebElement errorMessage;

    public void attemptLogin(String email, String password) {
        emailField.sendKeys(email);
        passwordField.sendKeys(password);
        loginButton.click();
    }

    public String getErrorMessage() {
        WebDriverWait wait = new WebDriverWait(
            Driver.getDriver(), Duration.ofSeconds(10));
        wait.until(ExpectedConditions.visibilityOf(errorMessage));
        return errorMessage.getText();
    }
}

Test Implementation:

public class FacebookLoginTest {

    @Test
    public void negativeLoginTest() {
        Driver.getDriver().get("https://www.facebook.com/");

        FacebookLoginPage loginPage = new FacebookLoginPage();
        Faker faker = new Faker();

        // Generate fake credentials
        String fakeEmail = faker.internet().emailAddress();
        String fakePassword = faker.internet().password();

        // Attempt login with fake credentials
        loginPage.attemptLogin(fakeEmail, fakePassword);

        // Verify error message appears
        String errorMsg = loginPage.getErrorMessage();

        Assert.assertTrue(
            errorMsg.contains("incorrect") ||
            errorMsg.contains("invalid"),
            "Error message should indicate login failure"
        );
    }

    @AfterMethod
    public void tearDown() {
        Driver.closeDriver();
    }
}

Important Considerations:

  • Dynamic Elements: Error messages may load dynamically - use explicit waits
  • Locator Strategy: Error message locators can change frequently
  • Faker Library: Useful for generating realistic test data
  • Negative Testing: Essential for robust test coverage

POM Best Practices

Do's:

  • Use meaningful page and method names
  • Keep WebElements in Page Classes
  • Use @FindBy annotations instead of findElement()
  • Initialize pages with PageFactory.initElements()
  • Create separate page class per web page
  • Use appropriate access modifiers
  • Implement Singleton pattern for Driver

Don'ts:

  • Don't put WebElements in Test Classes
  • Don't use driver.findElement() in tests
  • Don't forget to initialize Page Factory
  • Don't create overly complex page methods
  • Don't mix test logic with page logic
  • Don't hardcode test data in page classes

Framework Organization Tips:

Package Structure

Maintain clean separation: pages, tests, utilities

Naming Conventions

Use consistent naming: LoginPage, ProductsPage, etc.

Method Responsibilities

Page methods should perform actions, test methods should verify results

Configuration Management

Use properties files for URLs, credentials, and settings
