Cross Browser Testing

What is Cross Browser Testing?

Cross Browser Testing is a type of functional testing to check whether your web application works as expected in different browsers.

Browser Environment Reality:

  • Web-based applications are completely different from Windows applications
  • A web application can be opened by the end user in any browser
  • Some people prefer Firefox browser
  • Others may be using Chrome or Internet Explorer
  • Different browsers may render the same application differently

Selenium WebDriver Browser Support:

Chrome

ChromeDriver

Firefox

GeckoDriver

Edge

EdgeDriver

Safari

SafariDriver

IE

IEDriver

Why Cross Browser Testing?

  • User Experience: Ensure consistent experience across all browsers
  • Market Coverage: Reach maximum audience
  • Quality Assurance: Identify browser-specific issues
  • Business Impact: Prevent revenue loss due to browser incompatibility

XML Configuration for Cross Browser Testing

Cross Browser Testing XML Configuration:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Cross browser test">
    <test name="Cross browser chrome">
        <parameter name="browser" value="chrome"></parameter>
        <classes>
            <class name="cw.tests.day15_assertions_xmlFiles.CrossBrowserTest"></class>
        </classes>
    </test>
    <test name="Cross browser firefox">
        <parameter name="browser" value="firefox"></parameter>
        <classes>
            <class name="cw.tests.day15_assertions_xmlFiles.CrossBrowserTest"></class>
        </classes>
    </test>
</suite>

XML Structure Benefits:

  • Same Test Class: Run identical test logic
  • Different Parameters: Browser-specific configuration
  • Easy Management: Add/remove browsers easily
  • Parallel Execution: Run browsers simultaneously

Key Configuration Points:

  • Parameter Name: Use "browser" as parameter name
  • Parameter Value: Specify browser type (chrome, firefox, edge)
  • Same Class: Use same test class for all browsers
  • Different Tests: Each browser gets its own test tag

Execution Process:

  1. TestNG reads the XML configuration
  2. For each <test> tag, it passes the browser parameter
  3. The same test class runs with different browser configurations
  4. Each test executes independently with its specified browser

Cross Browser Test Implementation

Base Class with Browser Parameter Handling:

public class ParametersBrowserClass {
    public static WebDriver driver;

    @Parameters("browser")
    @BeforeMethod
    public void setup(@Optional("chrome") String browser) {
        // It is subject to switch or if-else blocks.
        if (driver == null) {
            switch (browser) {
                // If our browser is "chrome", chromedriver is set.
                case "chrome":
                    driver = new ChromeDriver();
                    break;
                // If our browser is "firefox", firefoxdriver is set.
                case "firefox":
                    driver = new FirefoxDriver();
                    break;
                case "edge":
                    driver = new EdgeDriver();
                    break;
            }
        }
        driver.manage().window().maximize();
        driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
    }
}

Key Implementation Features:

  • @Parameters("browser"): Receives browser parameter from XML
  • @Optional("chrome"): Default browser if parameter not provided
  • Switch Statement: Handle different browser types
  • Driver Management: Initialize appropriate driver

Important Considerations:

  • Driver Paths: Ensure driver executables are available
  • Version Compatibility: Match browser and driver versions
  • Thread Safety: Consider ThreadLocal for parallel execution
  • Resource Cleanup: Properly close drivers after tests

Enhanced Implementation with ThreadLocal:

private static ThreadLocal<WebDriver> driverThreadLocal = new ThreadLocal<>();

public static WebDriver getDriver() {
    return driverThreadLocal.get();
}

@AfterMethod
public void tearDown() {
    if (getDriver() != null) {
        getDriver().quit();
        driverThreadLocal.remove();
    }
}

Data Provider Introduction

What is Data Provider?

The DataProvider property is one of the important features TestNG provides. It is the second way to pass parameters to test methods.

Data-Driven Testing:

Data-driven testing allows users to run the same test method multiple times with different test data sets.

This approach enables comprehensive testing with various input combinations without duplicating test code.

DataProvider Characteristics:

  • Returns Array: Two-dimensional array of objects Object[][]
  • First Array: Represents a data set
  • Second Array: Contains parameter values
  • Location Flexibility: Can be in same class or superclass

Comparison with Other Approaches:

  • vs @Parameters: More flexible, supports multiple data sets
  • vs Cucumber: Similar to Scenario Outline functionality
  • vs Hard-coded data: Separates test logic from test data
  • vs External files: Data defined within code, easier debugging

DataProvider Benefits:

  • Code Reusability: One test method handles multiple data sets
  • Maintainability: Easy to add/remove test data
  • Comprehensive Testing: Test multiple scenarios efficiently
  • Clear Separation: Test logic separated from test data

DataProvider Annotation and Syntax

@DataProvider Annotation:

TestNG provides an annotation called @DataProvider to use the DataProvider property in our tests.

This annotation is declared in our test class with a method that can be called later in test methods.

Basic Syntax:

@DataProvider(name = "myTestData")
public Object[][] dataProviderMethod() {
    return new Object[][] {
        {"data1", "value1"},
        {"data2", "value2"},
        {"data3", "value3"}
    };
}

Annotation Attributes:

  • name: Optional name for the DataProvider
  • parallel: Enable parallel execution of data sets
  • indices: Specify which data sets to use

Method Requirements:

  • Return Type: Must return Object[][]
  • Access Modifier: Should be public
  • Static: Can be static for cross-class usage
  • Parameters: Can accept Method or ITestContext

Usage in Test Methods:

@Test(dataProvider = "myTestData")
public void testMethod(String data, String value) {
    // Test logic using the provided data
    System.out.println("Testing with: " + data + " and " + value);
}
  • dataProvider: Links to the DataProvider method name
  • Parameters: Method parameters match DataProvider return values
  • Execution: Test runs once for each data set

DataProvider Implementation Example

Complete DataProvider Example:

import org.testng.annotations.DataProvider;
import org.testng.annotations.Test;

public class DataProviderExample {
    // This test method is linked to the "getData" DataProvider.
    // It will be executed once for each row of data returned by the DataProvider.
    @Test(dataProvider = "getData")
    public void UserNameTest(String username, String password) {
        System.out.println("username : " + username + " password : " + password);
    }

    // This method acts as a data provider, supplying data to the test methods.
    // The data is a 2D array of Objects, where each inner array contains
    // the parameters for a single test run.
    @DataProvider
    public Object[][] getData() {
        Object[][] data = {
            {"John", "123"},
            {"Mike", "asd"},
            {"Peter", "1q2w"}
        };
        return data;
    }
}

Execution Flow:

DataProvider Method Called

TestNG calls getData() method to retrieve test data

Data Array Returned

Method returns 2D array with 3 rows of data

Test Method Execution

UserNameTest runs 3 times, once for each data row

Parameter Passing

Each execution receives different username/password values

Expected Output:

username : John password : 123
username : Mike password : asd
username : Peter password : 1q2w

Key Points:

  • Same Function as Cucumber: Similar to Scenario Outline functionality
  • Multiple Executions: One test method, multiple data sets
  • Parameter Mapping: Array elements map to method parameters in order
  • Data Types: Can use different data types (String, int, boolean, etc.)
