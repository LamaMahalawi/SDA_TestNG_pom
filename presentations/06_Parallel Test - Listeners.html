Parallel Testing

What is Parallel Testing?

Parallel testing is an approach where independent tests are run in parallel to reduce overall test execution time.

Key Concepts:

  • Multiple Versions: Running two or more versions of software simultaneously
  • Same Input: Using identical input or test methods across parallel executions
  • Cross-Platform: Tests can run on multiple devices, browsers, and operating systems
  • Flexible Scope: Can be applied to single applications, multiple applications, or components

Advantages of Parallel Testing:

  • Reduces Execution Time: Multiple tests executed simultaneously
  • Multi-threaded Testing: Enables verification of multi-threaded code
  • Cross-Browser Testing: Simultaneous execution on multiple browsers
  • Replaces Sequential Testing: More efficient than traditional sequential execution

Parallel Testing with XML File

Multi-Threading in TestNG

Multi-threading executes multiple threads simultaneously without interdependencies. Parallel testing in TestNG is configured through the XML file.

XML Configuration for Parallel Testing:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testing.org/testing-1.0.dtd">
<suite name="Parallel Testing Suite" parallel="methods" thread-count="2">
    <test name="Parallel Tests">
        <classes>
            <class name="clarusway.tests.Day05_C01_ParallelTest"/>
        </classes>
    </test>
</suite>

Configuration Parameters:

  • parallel: Defines the level of parallel execution (methods, tests, classes)
  • thread-count: Number of threads to use for parallel execution
  • Best Use: When running multiple tests that take a long time

Parallel Testing with Test Parameters

Method-Level Parallel Execution

TestNG allows configuring individual test methods to run in parallel using test parameters.

Test Parameters for Parallel Execution:

public class ParallelTestParameter {

    /**
     * This test method will be invoked multiple times in parallel threads
     * threadPoolSize specifies the number of threads to use
     * invocationCount specifies how many times to invoke the test method
     * The test will time out if all invocations don't complete within specified time
     */
    @Test(threadPoolSize = 4, invocationCount = 5, timeOut = 1000)
    public void testMethod() {
        System.out.println("Thread ID is : " + Thread.currentThread().getId());
    }
}

Key Parameters:

  • threadPoolSize: Number of threads to create for parallel execution
  • invocationCount: Number of times to invoke the test method
  • timeOut: Maximum time allowed for test execution (in milliseconds)

Parallel Testing with Data Provider

Data-Driven Parallel Execution

Data providers can be configured to run test iterations in parallel, either from class or XML configuration.

XML Configuration for Data Provider Parallel Testing:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testing.org/testing-1.0.dtd">
<suite name="Parallel Testing Suite" data-provider-thread-count="3">
    <test name="Parallel Tests">
        <classes>
            <class name="clarusway.tests.Day05_C04_ParallelTestDataProvider"/>
        </classes>
    </test>
</suite>

Data Provider with Parallel Execution:

public class Day05_C04_ParallelTestDataProvider {

    @Test(dataProvider = "data")
    public void personNames(String First_Name, String Last_Name) {
        System.out.println("Person Names: " + First_Name + " " + Last_Name);
    }

    // By default, parallel = false. To enable parallel execution, specify parallel = true
    @DataProvider(name = "data", parallel = true)
    public Object[][] myDataProvider() {
        Object data[][] = new Object[3][2];

        // First Person Names details
        data[0][0] = "John";
        data[0][1] = "Doe";

        // Second Person Names details
        data[1][0] = "Jane";
        data[1][1] = "Smith";

        // Third Person Names details
        data[2][0] = "Peter";
        data[2][1] = "Jones";

        return data;
    }
}

Parallel Execution with Maven

Maven Surefire Plugin Configuration

Parallel tests can be executed using mvn clean verify command with Maven Surefire Plugin configuration.

pom.xml Configuration:

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.8.1</version>
        </plugin>

        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-surefire-plugin</artifactId>
            <version>3.1.2</version>
            <configuration>
                <parallel>methods</parallel>
                <threadCount>10</threadCount>
                <redirectTestOutputToFile>true</redirectTestOutputToFile>
            </configuration>
        </plugin>
    </plugins>
</build>

Surefire Plugin Properties:

  • parallel: Set to "methods" for parallel method execution
  • threadCount: Number of threads for parallel execution
  • redirectTestOutputToFile: Redirects test output to files

Smoke Testing

What is Smoke Testing?

Smoke testing is a software testing process that determines whether the deployed software build is stable. It serves as an approval for the test team to proceed with further testing.

Key Characteristics:

  • Also Known As: Build verification testing or build acceptance testing
  • Non-Exhaustive: Verifies most important functions without finer details
  • Pre-Release Check: Conducted after build and before release
  • Problem Detection: Finds fundamental and critical problems early

Smoke Test Structure:

- [ ] tests
- [x] smokeTest
- [ ] NegativeLoginTest
- [ ] PositiveLoginTest

Smoke Test Implementation

Positive Login Test Example:

public class PositiveLoginTest extends TestBase {

    @Test
    public void test() {
        driver.get("https://practicetestautomation.com/practice-test-login/");
        driver.findElement(By.id("username")).sendKeys("student");
        driver.findElement(By.name("password")).sendKeys("Password123");
        driver.findElement(By.cssSelector("button#submit")).click();

        String expectedUrl = "https://practicetestautomation.com/logged-in-successfully/";
        String actualUrl = driver.getCurrentUrl();
        Assert.assertEquals(actualUrl, expectedUrl, "Actual page url is not the same as expected");

        WebElement successMessage = driver.findElement(By.tagName("strong"));
        String expectedMessage = "Congratulations student. You successfully logged in!";
        String actualMessage = successMessage.getText();
        Assert.assertTrue(actualMessage.contains(expectedMessage),
            "Actual message does not contain expected message.\nActual Message: " +
            actualMessage + "\nExpected Message: " + expectedMessage);

        WebElement logOutButton = driver.findElement(By.linkText("Log out"));
        Assert.assertTrue(logOutButton.isDisplayed(), "Log out button is not visible");
    }
}

Negative Login Test Example:

public class NegativeLoginTest extends TestBase {

    @Test
    public void test() {
        driver.get("https://practicetestautomation.com/practice-test-login/");
        driver.findElement(By.id("username")).sendKeys("student");
        driver.findElement(By.name("password")).sendKeys("incorrectPassword");
        driver.findElement(By.cssSelector("button#submit")).click();

        SoftAssert softAssert = new SoftAssert();
        String expectedErrorMes = "Your password is invalid!";
        WebElement actualErrorMes = driver.findElement(By.cssSelector("div[id='error']"));
        softAssert.assertTrue(actualErrorMes.isDisplayed());
        softAssert.assertEquals(actualErrorMes.getText(), expectedErrorMes);
        softAssert.assertAll();
    }
}

Smoke Test XML Configuration

XML Configuration for Smoke Tests:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testing.org/testing-1.0.dtd">
<suite name="Smoke test suite" verbose="5">
    <test name="Smoke Test">
        <packages>
            <package name="clarusway.tests.smokeTest"/>
        </packages>
    </test>
</suite>

XML Configuration Notes:

  • verbose: Controls the amount of information printed in the terminal
  • packages: Allows grouping tests by package structure
  • Range: Verbose levels range from 0-10, with 5 providing moderate detail

TestNG Listeners

What are TestNG Listeners?

Listeners are used to change the default behavior of TestNG reporting and logs. A TestNG listener is code that listens for events occurring in TestNG and executes custom code in response.

Types of TestNG Listeners:

  • ITestListener - Most commonly used listener interface
  • IReporter - For custom reporting
  • ISuiteListener - For suite-level events
  • IHookable - For custom method invocation
  • IExecutionListener - For test execution events
  • IConfigurable - For configuration methods
  • IConfigurableListener - For configuration listener

ITestResult Interface:

public interface ITestResult
    extends IAttributes, java.lang.Comparable<ITestResult>

ITestListener Interface

ITestListener Overview

ITestListener is an interface that needs to be implemented in a class. It provides methods to hook into different test execution phases.

Key ITestListener Methods:

  • onStart - Called before any test method is executed
  • onFinish - Called after all test methods are executed
  • onTestStart - Called before any test method is called
  • onTestSkipped - Called when any test method is skipped
  • onTestSuccess - Called when any test method succeeds
  • onTestFailure - Called when any test method fails

Listener Implementation Example:

@Listeners(ListenerClass.class)
public class UseListenerClass {
    WebDriver driver;

    @Test
    public void testListener() {
        driver = new ChromeDriver();
        driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
        driver.manage().window().maximize();
        driver.get("https://translate.google.com/");

        driver.findElement(By.xpath("//*[@aria-label='Source text']")).sendKeys("lovely", Keys.ENTER);
        Assert.assertTrue(driver.findElement(By.xpath("//*[contains(text(),'lovely')]")).isDisplayed());
    }
}

Listener Implementation

Complete Listener Class:

public class ListenerClass implements ITestListener {

    public void onStart(ITestContext context) {
        System.out.println("onStart method started");
    }

    public void onFinish(ITestContext context) {
        System.out.println("onFinish method started");
    }

    public void onTestStart(ITestResult result) {
        System.out.println("New Test Started: " + result.getName());
    }

    public void onTestSuccess(ITestResult result) {
        System.out.println("onTestSuccess Method: " + result.getName());
    }

    public void onTestFailure(ITestResult result) {
        System.out.println("onTestFailure Method: " + result.getName());
    }

    public void onTestSkipped(ITestResult result) {
        System.out.println("onTestSkipped Method: " + result.getName());
    }

    public void onTestFailedButWithinSuccessPercentage(ITestResult result) {
        System.out.println("onTestFailedButWithinSuccessPercentage: " + result.getName());
    }
}

Implementation Notes:

  • Implement ITestListener interface
  • Override required methods
  • Use @Listeners annotation or XML configuration
  • Access test information through ITestResult parameter

ITestContext and ITestResult

ITestContext Interface

ITestContext provides methods for accessing information about the current test execution environment. This information can be used for logging, statistics, and customizing test behavior.

ITestContext Usage:

@Test
public void test1(ITestContext context) {
    // Access test context information
    // Sharing data across test methods
}

ITestResult Interface

ITestResult provides detailed information about individual test method execution, including status, execution time, failures, and skipped steps.

ITestResult Usage:

@Test
public void myTest(ITestResult result) {
    // Access test result information
}

Adding Listeners to HTML Report

Step 1: Access Run Configuration

Go to Edit configuration from the Run Menu in your IDE

Step 2: Select TestNG Configuration

In the left section, find your running module under the TestNG heading

Step 3: Add Listeners

Click Listeners from the right section, then click the + icon

Step 4: Select Report Listeners

Search for "report" and select "EmailableReporter" and "FailedReporter"

Step 5: Run Tests

Run the test again from the XML file to generate reports

Generated Reports:

  • test-output/report/ - HTML test reports
  • emailable-report.html - Email-friendly report
  • testng-failed.xml - XML for failed tests only
