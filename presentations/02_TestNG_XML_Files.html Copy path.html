Advanced Test Management with XML

XML Files Introduction

What is XML?

XML (Extensible Markup Language) is a markup language used to create documents that can be easily read by both humans and computer systems. It is a standard defined by the W3C.

XML is developed to standardize the exchange of data between platforms and systems that exchange data over the internet.

Main Advantages of TestNG.xml:

  • Allows parallel execution of test methods
  • Allows dependency of one test method on another
  • Helps prioritize our test methods
  • Allows test methods to be grouped into test groups
  • Supports parameterization using @Parameters annotation
  • Helps in data-driven testing using @DataProvider
  • Provides different types of assertions for validation
  • Generates HTML reports and coverage reports

TestNG XML Configuration File

What is TestNG.xml file?

The TestNG.xml file is a configuration file that helps us organize our tests.

  • Allows testers to create and manipulate multiple test classes
  • Define test suites and tests
  • Makes tester's job easier by controlling test execution
  • Brings all test cases together and runs them under a single XML file

Benefits of XML Configuration:

  • Centralized Control: Manage all tests from one location
  • Flexible Execution: Run specific tests, classes, or packages
  • Easy Maintenance: Modify test execution without code changes
  • Parallel Testing: Execute multiple tests simultaneously
  • Group Management: Run specific test groups
  • Parameterization: Pass parameters to tests

Documentation Reference:

For detailed information: https://testng.org/doc/documentation-main.html#testng-listeners

Running Specific Class in XML

Single Class Execution:

This way only one class is executed, but if we want to run more than one class, package, or method at the same time, we can use XML file.

Basic XML Structure for Single Class:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Suite1">
    <test name="test1">
        <classes>
            <class name="clarusway.tests.Day02_C4_XMLFiles01"></class>
        </classes>
    </test>
</suite>

Execution Process:

Use <classes> Element

Instead of <package> attribute, we use <classes> and then <class> attribute

Specify Class Name

Provide full package path: package.classname

Run the Test

Right-click on XML file and run the test

Key Points:

  • We can run the classes by right-clicking on the XML
  • Use full qualified class names (package.classname)
  • Each class runs all its @Test methods

Running Multiple Classes in XML

Multiple Classes in One Test:

Run multiple classes in one test at the same time

Multiple Classes in Same Test:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<!-- This section includes all the tests. -->
<suite name="Suite1">
    <test name="test1">
        <!-- This section specifies the classes that will be run. -->
        <classes>
            <class name="clarusway.tests.Day02_C4_XMLFiles01"></class>
            <class name="clarusway.tests.Day02_C5_XMLFiles02"></class>
            <class name="clarusway.tests.Day02_C6_XMLFiles03"></class>
        </classes>
    </test>
</suite>

Multiple Classes in Different Tests:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Suite1">
    <test name="test1">
        <classes>
            <class name="clarusway.tests.Day02_C4_XMLFiles01"></class>
        </classes>
    </test>
    <test name="test2">
        <classes>
            <class name="clarusway.tests.Day02_C5_XMLFiles02"></class>
        </classes>
    </test>
</suite>

Execution Benefits:

  • Same Test: Classes run sequentially within single test context
  • Different Tests: Each test has its own execution context
  • Flexibility: Mix and match classes as needed

XML Elements Overview

<suite>

Top-level container for all tests

<test>

Groups classes and methods together

<classes>

Container for class definitions

<class>

Specifies individual test class

<methods>

Include/exclude specific methods

<packages>

Run entire packages

<groups>

Execute grouped tests

<parameter>

Pass parameters to tests

XML Structure Hierarchy:

  1. suite (top level)
  2. ├── test
  3. │ ├── groups
  4. │ ├── classes
  5. │ │ └── class
  6. │ │ └── methods
  7. │ └── packages
  8. │ └── package
  9. └── parameter

Running Groups in XML

Group Execution:

To run a group, we must both define the group name and specify where to look for it.

Single Group Execution:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Suite1">
    <test name="test1">
        <!-- This section specifies the groups to be run. -->
        <groups>
            <run>
                <include name="SmokeTest"></include>
            </run>
        </groups>
        <!-- This section specifies the classes that will be run. -->
        <classes>
            <class name="clarusway.tests.Day02_C6_XMLFiles03"></class>
        </classes>
    </test>
</suite>

Multiple Groups Execution:

<suite name="Suite1">
    <test name="test1">
        <groups>
            <run>
                <include name="SmokeTest"></include>
            </run>
        </groups>
        <classes>
            <class name="clarusway.tests.Day02_C6_XMLFiles03"></class>
        </classes>
    </test>
    <test name="test2">
        <groups>
            <run>
                <include name="Regression"></include>
            </run>
        </groups>
        <classes>
            <class name="clarusway.tests.Day02_C6_XMLFiles03"></class>
        </classes>
    </test>
</suite>

Include/Exclude Groups in XML

Include and Exclude Groups:

<suite name="Suite1">
    <test name="test1">
        <groups>
            <run>
                <!-- Run only the Regression group -->
                <include name="Regression"></include>
                <!-- Exclude the SmokeTest group from being run -->
                <exclude name="SmokeTest"></exclude>
            </run>
        </groups>
        <classes>
            <class name="clarusway.tests.Day02_C6_XMLFiles03"></class>
        </classes>
    </test>
</suite>

<include> Element:

  • Specifies which groups to run
  • Multiple includes possible
  • Only specified groups execute

<exclude> Element:

  • Specifies which groups to skip
  • Takes precedence over include
  • Useful for temporarily disabling groups

Group Management Best Practices:

  • Logical Grouping: Create meaningful group names like "SmokeTest", "Regression"
  • Combine Include/Exclude: Use both for fine-grained control
  • Test Organization: Group related functionality together
  • Execution Control: Enable selective test execution for different environments

Include/Exclude Methods in XML

Method-Level Control:

<suite name="Suite1">
    <test name="test1">
        <groups>
            <run>
                <include name="SmokeTest"></include>
            </run>
        </groups>
        <classes>
            <class name="clarusway.tests.Day02_C6_XMLFiles03">
                <methods>
                    <!-- Exclude a specific method from the class -->
                    <exclude name="Test3"></exclude>
                </methods>
            </class>
        </classes>
    </test>
</suite>

Method Inclusion:

  • Use <include name="methodName">
  • Only specified methods will run
  • Override class-level execution
  • Useful for selective testing

Method Exclusion:

  • Use <exclude name="methodName">
  • Skip specific methods
  • All other methods in class run
  • Useful for temporary disabling

Combined Group and Method Control:

This example shows how to:

  • Include only SmokeTest group methods
  • Exclude specific method Test3 from the class
  • Provide fine-grained control over test execution

Use Cases:

  • Debugging: Run only problematic methods
  • Development: Skip long-running tests temporarily
  • Maintenance: Exclude broken tests while fixing

Running Packages in XML

Package Execution:

Runs all tests in the specified package. This is useful when you want to execute all test classes within a package.

Package Execution Example:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Suite1">
    <test name="test1">
        <!-- This section specifies the packages to be run.
             All classes within the specified package will be executed. -->
        <packages>
            <package name="clarusway.tests"></package>
        </packages>
    </test>
</suite>

Package Benefits:

  • Comprehensive: Runs all classes in package
  • Maintenance: No need to list individual classes
  • Scalable: New classes automatically included
  • Organization: Tests by functional area

Package Considerations:

  • All Inclusive: Runs ALL classes in package
  • No Filtering: Cannot exclude specific classes
  • Performance: May run more tests than needed
  • Dependencies: Ensure all classes are runnable

Best Practices for Package Execution:

  • Package Structure: Organize tests logically by functionality
  • Naming Convention: Use clear package names (e.g., login, checkout, reports)
  • Test Readiness: Ensure all classes in package are properly configured
  • Multiple Packages: Can specify multiple <package> elements

XML Parameters

XML Parameterization:

Parameters can be given to the test to be run from XML and the test can be executed from the XML file.

Parameter Passing Example:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<!-- This suite runs tests that use a parameterized search keyword. -->
<suite name="Suite1">
    <test name="test1">
        <!-- Pass 'java' as the search keyword. -->
        <parameter name="searchKeyword" value="java"></parameter>
        <classes>
            <class name="clarusway.tests.Day03_C7_XMLFilesParameter"></class>
        </classes>
    </test>
    <test name="test2">
        <!-- Pass 'selenium' as the search keyword. -->
        <parameter name="searchKeyword" value="selenium"></parameter>
        <classes>
            <class name="clarusway.tests.Day03_C7_XMLFilesParameter"></class>
        </classes>
    </test>
</suite>

Parameter Usage:

  • @Parameters annotation: Receive parameters in test methods
  • Dynamic values: Change test behavior without code changes
  • Environment specific: Different values for different environments

Java Code Example:

@Test
@Parameters("searchKeyword")
public void searchTest(String keyword) {
    // Use the keyword parameter
    driver.findElement(By.id("search"))
          .sendKeys(keyword);
}

Parameter Benefits:

  • Flexibility: Same test method with different data
  • Reusability: One test class, multiple configurations
  • Maintenance: Change test data without code modification
  • Environment Testing: Test different environments with different parameters

Advanced XML Features

Parallel Execution

Run tests in parallel for faster execution

parallel="methods"

Thread Count

Control number of parallel threads

thread-count="3"

Data Providers

External data sources for tests

@DataProvider

Listeners

Custom behavior during test execution

<listeners>

Parallel Execution Example:

<suite name="ParallelSuite" parallel="methods" thread-count="3">
    <test name="ParallelTest">
        <classes>
            <class name="tests.LoginTest"></class>
            <class name="tests.SearchTest"></class>
            <class name="tests.CheckoutTest"></class>
        </classes>
    </test>
</suite>

Parallel Options:

  • parallel="methods": Run test methods in parallel
  • parallel="classes": Run test classes in parallel
  • parallel="tests": Run test tags in parallel

Performance Considerations:

  • Thread Safety: Ensure tests are thread-safe
  • Resource Management: Manage shared resources carefully
  • Browser Instances: Use ThreadLocal for WebDriver

XML Configuration Best Practices

Do's:

  • Use meaningful suite and test names
  • Add comments to explain complex configurations
  • Group related tests logically
  • Use parameters for environment-specific values
  • Organize tests by functionality or priority
  • Keep XML files version controlled

Don'ts:

  • Don't create overly complex XML structures
  • Don't hardcode environment-specific values
  • Don't mix different types of tests inappropriately
  • Don't forget XML validation
  • Don't use excessive nesting

File Organization:

Separate XML Files

Create separate XML files for different test suites (smoke.xml, regression.xml)

Environment Specific

Use different XML files for different environments (dev.xml, staging.xml)

Clear Naming

Use descriptive names that indicate purpose and scope

Maintenance Tips:

  • Regular Review: Periodically review and update XML configurations
  • Documentation: Document complex configurations and their purposes
  • Validation: Use XML validators to ensure syntax correctness
  • Version Control: Track changes to XML files in version control

Common XML Patterns

Smoke Test Suite:

<suite name="SmokeTestSuite">
    <test name="CriticalFunctionality">
        <groups>
            <run>
                <include name="smoke"></include>
            </run>
        </groups>
        <packages>
            <package name="tests"></package>
        </packages>
    </test>
</suite>

Regression Test Suite:

<suite name="RegressionSuite" parallel="classes" thread-count="2">
    <test name="FullRegression">
        <groups>
            <run>
                <include name="regression"></include>
                <exclude name="slow"></exclude>
            </run>
        </groups>
        <packages>
            <package name="tests"></package>
        </packages>
    </test>
</suite>

Cross-Browser Testing:

<suite name="CrossBrowserSuite">
    <test name="ChromeTests">
        <parameter name="browser" value="chrome"></parameter>
        <classes>
            <class name="tests.LoginTest"></class>
        </classes>
    </test>
    <test name="FirefoxTests">
        <parameter name="browser" value="firefox"></parameter>
        <classes>
            <class name="tests.LoginTest"></class>
        </classes>
    </test>
</suite>

Troubleshooting XML Issues

Common Issues:

  • XML Syntax Errors: Missing tags, incorrect nesting
  • Class Not Found: Incorrect class path or name
  • Group Not Found: Group name mismatch
  • Parameter Mismatch: Parameter name doesn't match @Parameters
  • DTD Declaration: Missing or incorrect DOCTYPE

Solutions:

  • Validate XML: Use XML validators
  • Check Paths: Verify full package.class names
  • Match Groups: Ensure group names match exactly
  • Parameter Names: Match @Parameters annotation
  • DTD Reference: Use correct TestNG DTD URL

Debug Steps:

Validate XML syntax using IDE or online validators
